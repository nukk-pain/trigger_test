# ADR 0001: 전신 클릭맵 → 2단계(그룹→세부) 부위 선택 UI 전환

- 상태: 채택
- 날짜: 2026-06-05
- 관련: `.tasks/260605-bodymap-3step/PLAN.md`, Oracle gpt-5.5-pro 패널(/tmp/oracle-bodymap.md, 17 sources)

## 배경

통증 셀프마사지 가이드의 부위 선택은 전신 SVG에 96개 클릭 핫스팟을 깔아 두었다. 두 가지 문제가 있었다.

1. **False affordance**: MVP가 실제 분석을 제공하는 부위는 목·어깨·어깨뼈·등 상부/중부·허리·골반·엉덩이·천골(3그룹)뿐인데, 팔·다리·머리·가슴·복부 등 미지원 부위까지 전부 클릭 가능해 보였다. 누르면 "준비 중" 안내만 떠 신뢰를 깎았다.
2. **모바일 탭타깃**: 96개가 한 화면에 빽빽해 작은 부위는 16~24px로, WCAG/Android 권장(44~48px)에 한참 못 미쳤다.
3. **마네킹 시각**: 도형(원/사각형) 조합으로 만든 실루엣이 사람으로 안 읽혔다.

## 결정

전신 클릭맵을 폐기하고 **2단계 progressive disclosure**로 전환한다.

- **1단계**: 큰 부위군 3개(목·어깨 / 등·허리 / 골반·엉덩이)만 선택 가능. 후면 중심 단일 path 실루엣을 배경으로 두고, 그룹 3개만 강조. 같은 선택지를 48px 리스트 카드로 병행(리스트가 primary, 그림은 shortcut).
- **2단계**: 선택한 그룹의 **실제 지원 세부 data-area를 한글명 버튼**으로 직접 노출(다중 선택). 좌/우/중앙은 별도 단계로 쪼개지 않는다.

분석 계약은 보존한다: 최종 선택은 기존과 동일한 세부 `data-area` 문자열을 `selectedAreas`에 넣어, `analyzeTriggerPoints`(직접 매칭)와 `analyzeFascialLines`(mapAreaToGroup 변환) 둘 다 그대로 동작한다.

## 대안

1. **전신 클릭맵 유지 + 미지원 숨김**: 변경 최소. 그러나 모바일 탭타깃·마네킹 문제는 그대로. 기각.
2. **3단계(그룹 → 세부 → 좌/우/양쪽/가운데 segmented control)**: Oracle 1차 권고. 그러나 사전 코드 검증에서 지원 부위의 side 변형이 **불규칙**함이 드러났다 — `neck-front`/`mid-back-center`/`sacral`은 좌우가 없고, `buttock`은 좌/우 + 상/하로 분화. side 단계를 일률 적용하면 잘못된 data-area가 분석에 들어가 **잘못된 의료 가이드**로 이어진다. 기각.
3. **2단계(그룹 → 세부 한글명 버튼) [채택]**: 세부 부위의 한글명("허리 중앙", "왼쪽 엉덩이 위")이 `getAreaDisplayName`에 이미 모두 존재하므로, side를 추론·환원하지 않고 버튼 라벨로 직접 노출한다. 매핑 오류 위험이 원천 제거되고 구현도 단순.

## 결과

- **긍정**: 정직한 affordance(지원 부위만), 모바일 탭타깃 충족, 분석 계약 무손상, 매핑 오류 0.
- **비용(감수)**: 전신 클릭맵 전제의 기존 단위 테스트(`selection-ui`, `selection-events`)와 e2e 동선을 **재작성**한다. selection-events/ui/renderer의 역할을 신규 `region-select`/`region-flow`/`area-groups`로 이관한다. 이는 비가역적이며, 그래서 이 ADR로 *왜*를 남긴다.
- **후속**: 그룹별 세부 버튼이 많은 경우(목·어깨 13개) 2차 소그룹핑 여지. '다른 부위가 불편해요' 피드백의 서버 수집은 별도 기능.
